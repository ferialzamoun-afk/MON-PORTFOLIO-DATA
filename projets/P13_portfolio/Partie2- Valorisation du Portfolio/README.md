# Mission 2 - Realisation du portfolio

## Objectif

La Mission 2 consiste a transformer les livrables du parcours Data Analyst en portfolio professionnel publie sur GitHub Pages.

Le portfolio n'est pas seulement une liste de projets : il doit raconter des preuves de competences lisibles par un jury, un recruteur ou un client.

## Narration des preuves de competences

Chaque projet ou preuve doit suivre le fil suivant :

| Etape | Contenu attendu |
|---|---|
| Contexte | Situer le projet, l'environnement et l'utilisateur cible. |
| Besoin metier | Expliquer la question ou la decision a aider. |
| Demarche | Montrer les donnees, les methodes, les outils et les arbitrages. |
| Resultats | Donner les KPI, livrables, captures et liens publics. |
| Impact | Expliquer la valeur produite et les limites. |

## Integration des resultats de veille

La veille est integree dans le portfolio pour justifier les choix :

- choix des outils de publication : GitHub Pages, Streamlit, nbviewer ;
- choix des methodes d'analyse : controles qualite, visualisation, statistiques, IA ;
- choix de presentation : preuve courte pour recruteur, preuve detaillee pour jury ;
- limites : donnees sensibles, liens publics, perimetre des projets.

## Correction vs evolution

| Type | Definition | Exemple portfolio |
|---|---|---|
| Correction | Reparer un probleme qui casse la lecture ou la fiabilite. | Lien mort, tableau mal rendu, chiffre obsolete, notebook non aligne. |
| Evolution | Ajouter de la valeur sans corriger un blocage. | Nouvelle capture, nouvelle fiche projet, enrichissement d'une preuve. |

Avant une soutenance, les corrections passent avant les evolutions.

## TNR GitHub Pages

Avant chaque publication importante, je controle :

- build Jekyll local sans erreur ;
- pages publiques `.html` accessibles ;
- liens GitHub, Streamlit et nbviewer fonctionnels ;
- pas de tableaux Markdown rendus en texte brut ;
- coherence entre Partie 1 et Partie 2 ;
- absence de donnees sensibles ou chemins nominatifs.
